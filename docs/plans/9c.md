# 9c — secrets: vaults & unlock

ROADMAP 9c: multiple vaults (vault_id + per-vault master key), Travel Mode (subset flag filters
list/reveal), biometric (WebAuthn platform authenticator) unlock.

## Audit (current state)
- Single global vault. `vault_verifier` in settings.json verifies the one master pw; entries all live
  in `vault_entries` with no vault scoping. `routes/vault.py` `_unlock_tokens` maps token → (exp, pw),
  resolved by `_master_pw`. 9a/9b ride on this (totp, watchtower, attachments, share).
- No per-vault concept, no travel mode, no webauthn. `services/crypto.py` is master-pw AES-GCM only.

## 9c-1 — multiple vaults + Travel Mode (unittest) — `tests/test_vault_multi.py`
Files: `core/database.py` (`Vault` model + `VaultEntry.vault_id` col), `routes/vault.py`,
`core/settings.py` (none — travel flag lives in settings via load/save).
- `Vault` row: id, name, verifier, travel_safe (bool), created_at. A lazily-created **default** vault
  (id `"default"`) absorbs the legacy `vault_verifier` setting and any pre-existing entries (vault_id
  NULL → default), so existing unlock/CRUD stays backward-compatible.
- Unlock is per-vault: `POST /vault/unlock {password, vault_id?}` → token bound to (pw, vault_id);
  no vault_id ⇒ default. `_ctx` resolves token→(pw, vault_id); `_master_pw` stays (pw only).
- `GET /vault/vaults` (list: id/name/travel_safe/entries), `POST /vault/vaults {name,password}`,
  `PATCH /vault/vaults/{vid} {name?,travel_safe?}`, `DELETE /vault/vaults/{vid}` (not default).
- `list_vault` + `watchtower` scope by the token's vault_id (cross-vault isolation).
- Travel Mode: `vault_travel_mode` setting + `PUT /vault/travel-mode {on}` /
  `GET /vault/travel-mode`. When on, `/vault/vaults` lists only travel_safe vaults and unlocking a
  non-safe vault is refused.
Tests (≥8): default_vault_exists, create_vault, vaults_list_shape, entries_scoped_to_vault,
cross_vault_isolation, unlock_specific_vault, rename_and_flag_vault, delete_vault, cannot_delete_default,
travel_mode_lists_only_safe, travel_mode_blocks_unsafe_unlock, legacy_verifier_migrates.

## 9c-2 — WebAuthn biometric unlock (unittest) — `tests/test_vault_webauthn.py`
Files: new `services/webauthn.py`, `core/database.py` (`WebAuthnCredential`), `routes/vault.py`,
`services/crypto.py` (no change — reuse encrypt/decrypt for the biometric blob).
- `services/webauthn.py`: `new_challenge()` (random, b64url) + `verify_assertion(public_key_der,
  authenticator_data, client_data_json, signature, challenge)` — checks clientData type=="webauthn.get"
  + challenge match, rebuilds `authData || sha256(clientDataJSON)`, verifies the ECDSA-P256 signature
  against the stored SPKI public key (the browser ships it via `getPublicKey()`, so no CBOR parsing).
- `WebAuthnCredential` row: id, vault_id, label, credential_id, public_key (b64 DER), sign_count.
- Enabling biometric requires an active unlock; the master pw is wrapped (`crypto.encrypt`) under a
  per-install server key kept in settings (`vault_biometric_key`) and stored per vault
  (`vault_biometric_blob`), so a verified assertion can release an unlock token.
- `POST /vault/webauthn/register` (unlock-gated), `GET /vault/webauthn/credentials`,
  `DELETE /vault/webauthn/credentials/{id}`, `GET /vault/webauthn/challenge?vault_id=`,
  `POST /vault/webauthn/unlock {vault_id, credential_id, authenticator_data, client_data_json,
  signature}` → mints an unlock token on a valid assertion.
Tests (≥8): challenge_random, verify_valid_assertion, verify_bad_signature, verify_wrong_challenge,
verify_wrong_type, register_requires_unlock, register_stores_credential, list_credentials,
delete_credential, webauthn_unlock_returns_token, webauthn_unlock_bad_assertion_401,
biometric_blob_not_plaintext.
