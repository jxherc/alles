# 9d — secrets: passkeys & autofill

ROADMAP 9d: passkey storage, hardware-key (FIDO2/YubiKey) 2FA for unlock, local browser-extension
autofill (a thin extension talking to the localhost vault API).

## Audit (current state, from 9c)
- Vault now multi-vault with WebAuthn assertion verify (`services/webauthn.py`) + biometric unlock.
- No passkey items (the vault can't hold a passkey it presents to a site). No 2FA gate on unlock
  (biometric is an *alternative* to the password, not a second factor). No autofill surface / extension.

## 9d-1 — passkey storage + use (unittest) — `tests/test_vault_passkey.py`
Files: new `services/passkey.py`, `routes/vault.py`.
- A passkey vault item (`type: "passkey"`): encrypted fields hold `rp_id`, `username`,
  `credential_id` (b64), `private_key` (PKCS8 PEM). `services/passkey.py`:
  `create_passkey(rp_id, username)` → generates an ES256 keypair, returns
  `{credential_id, public_key (SPKI b64), private_key_pem, rp_id, username}`;
  `sign(private_key_pem, authenticator_data, client_data_json)` → ES256 signature over
  `authData || sha256(clientDataJSON)` (round-trips through `webauthn.verify_assertion`).
- `POST /api/vault/passkey/new {rp_id, username}` (unlock-gated) creates the entry, returns the
  public credential (never the private key). `GET /api/vault/passkeys` lists passkey items
  (id/name/rp_id/username). `POST /api/vault/{id}/passkey/sign {authenticator_data, client_data_json}`
  signs an assertion with the stored private key.
Tests (≥8): generate_keypair, create_returns_public_credential, private_key_encrypted_in_entry,
sign_roundtrips_with_verify, list_passkeys, passkey_entry_has_rpid, create_requires_unlock,
sign_unknown_404, two_passkeys_distinct_keys, sign_requires_unlock.

## 9d-2 — hardware-key 2FA gate for unlock (unittest) — `tests/test_vault_hwkey.py`
Files: `core/database.py` (`WebAuthnCredential.role`), `routes/vault.py`.
- Register a *cross-platform* (security-key) credential with `role="2fa"`. Per-vault setting
  `require_2fa` (in settings, keyed by vault). When on, `POST /vault/unlock` with a correct password
  does NOT return a token — it returns `{requires_2fa: true, challenge, credentials}`. A second call
  `POST /vault/unlock/2fa {vault_id, password, credential_id, authenticator_data, client_data_json,
  signature}` verifies the password AND the assertion before minting a token.
Tests (≥8): register_2fa_credential, enable_2fa_requires_unlock, unlock_returns_challenge_when_2fa,
unlock_withholds_token_when_2fa, twofa_unlock_valid, twofa_wrong_password_401, twofa_bad_assertion_401,
twofa_unknown_credential_404, disable_2fa_restores_plain_unlock, twofa_status_in_settings.

## 9d-3 — browser-extension autofill (unittest + files) — `tests/test_vault_autofill.py`
Files: `routes/vault.py`, new `extension/manifest.json`, `extension/content.js`,
`extension/background.js`, `extension/popup.html`.
- `GET /api/vault/match?domain=` (unlock-gated): login-type entries whose stored `url` host matches
  `domain` (www-stripped, subdomain-aware), returning `{id, name, username, password}` for autofill.
- A thin MV3 extension under `extension/` that asks the localhost vault API for matches and fills a
  login form (packaging is a separate deliverable; we ship the source + a manifest).
Tests (≥8): match_by_host, match_strips_www, match_subdomain, no_match_empty, match_requires_unlock,
match_returns_credentials, match_only_logins, extension_manifest_valid_mv3, extension_has_content_script,
match_case_insensitive.
