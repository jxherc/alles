# 7c — gallery: share & motion

ROADMAP 7c: shared albums (1a), Live-Photo/video assets, phone-camera backup watch-folder.

## Audit (current state)
- 1a share primitive (`services/share.py`) already supports kind `album` in `VALID_KINDS`; generic
  `POST/GET/DELETE /api/share` already mint/lookup/revoke an album token. But `/s/{token}` only renders
  doc/file/folder/photo — **no album viewer** (line "album/contact/event ... land here in their own stages").
- No **video** support at all: `photos_store._ALLOWED` is images only; no `is_video`; lightbox is `<img>`.
- **No ffmpeg** on this box → video posters are adapted (no server-side frame extraction; grid shows a
  ▶ badge, lightbox plays via `<video controls>`).
- `photo_sync.sync_folder` does a manual folder import; there is **no watch-folder job**.

## 7c-1 — shared albums (unittest) — `tests/test_share_album.py`
Files: `routes/shared.py`.
- `/s/{token}` kind `album` → read-only HTML grid of the album's photos (excl hidden + deleted); images
  load via the confined child route.
- `/s/{token}/{photo_id}` kind `album` → serve that photo's image only if it belongs to the album and is
  live (not hidden/deleted); otherwise 404.
Tests (>=8): mint via /api/share, viewer renders grid, viewer 404 unknown/revoked, child serves member,
child 404 non-member, child 404 hidden, child 404 deleted, grid excludes hidden/deleted, empty album ok.

## 7c-2 — video assets (unittest) — `tests/test_photos_video.py`

  no PIL decode (dims 0, thumb ""), taken_at = now.
- `_fmt` carries `is_video` (+ keep `original` as the playback URL). `/list`/`/search` include videos.
Tests (>=8): import_media_video_flag, import_media_image_still_works, unsupported_ext_rejected,
upload_video_sets_is_video, fmt_has_is_video, list_includes_video, search_matches_video_name,
video_original_served, image_not_flagged_video.
